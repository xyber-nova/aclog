## Context

当前终端交互实现几乎全部堆在 `src/tui.rs` 中，既包含 alternate screen 生命周期，也包含 selector、sync、browse、stats 多类页面的绘制与键位处理。这样的组织使共享样式难以复用，页面之间的阅读顺序不一致，新增视觉层级或帮助机制时也需要在多个局部逻辑里重复修改。

本次改动是明显的跨模块变化：应用层仍然通过 `UserInterface` trait 与界面交互，但真实终端实现需要从单文件迁移到 `src/ui/terminal/`，同时对多个页面的布局、颜色和键位提示做一致性升级。项目已有大量 fake UI 和 workflow 测试，因此新设计必须保留现有应用层接口，并把变化限制在真实终端实现内部。

## Goals / Non-Goals

**Goals:**
- 将真实终端实现按页面与共享能力拆分到 `src/ui/terminal/`，避免继续在 `src/tui.rs` 中集中膨胀。
- 为 sync、selector、browser、stats 建立统一的训练工作台设计语言、状态颜色、选中态和帮助提示约定。
- 让主要交互页面形成稳定的阅读结构：顶部上下文、主体列表/内容区、右侧摘要/详情区、底部操作提示。
- 在不破坏现有命令语义和 `UserInterface` trait 的前提下，统一 `Enter / Esc / q / Tab` 工作流语义，并增加 `j/k` 导航别名和 `?` 帮助切换。
- 保持 `src/tui.rs` 作为兼容层，避免立即打破库导出的现有路径。

**Non-Goals:**
- 不引入搜索输入框、内联过滤编辑、鼠标交互或新的终端依赖。
- 不改变 app/domain 层的数据模型，也不改变 CLI 命令参数与非交互语义。
- 不在本次改动中移除 `src/tui.rs` 的兼容导出。
- 不在本次改动中引入 `onboard` / 新手教程命令或独立教学界面。

## Decisions

### 1. 保留 `UserInterface` 抽象，只重构真实终端实现
应用层和测试层已经稳定依赖 `src/ui/interaction.rs` 中的 `UserInterface` 与 `FakeUi` 语义。保留这层接口可以让 workflow 测试完全不感知 TUI 目录重组，也符合“交互抽象与 TUI 适配分离”的既有架构。

备选方案：
- 把 trait 和终端实现合并回单一模块。这样目录更少，但会让 app 层重新直接依赖具体终端实现，削弱测试替身边界。
- 彻底去掉 `src/tui.rs`。这样结构更纯，但会破坏当前公开模块路径，兼容成本高于本次收益。

### 2. 新增 `src/ui/terminal/` 模块树，并由 `src/tui.rs` 转发
真实终端实现拆为 `mod.rs`、`theme.rs`、`common.rs`、`sync.rs`、`selector.rs`、`browser.rs`、`stats.rs`。`mod.rs` 负责 `run_in_terminal` 和对外入口；各页面模块只负责自己的状态、绘制和事件循环；`theme/common` 提供共享样式和布局 helper。

备选方案：
- 按“组件类型”拆分，例如所有 helper 全放 `widgets.rs`。这种拆法对当前页面级工作流不友好，后续维护时仍需跨多个文件追踪同一页面。
- 仅把 `src/tui.rs` 拆成两个大文件。能稍微缓解体量，但无法建立共享主题和页面边界。

### 3. 共享主题采用“语义色 + 克制高亮”的终端视觉方案
主题层集中定义 verdict、warning、invalid、accent、hint、selected 等样式，避免页面各自硬编码颜色。选中态使用“指示符 + 背景/前景高亮 + 粗体”，替代只靠 `REVERSED` 的弱区分方式。

备选方案：
- 每个页面单独配色。短期灵活，但会快速失去一致性。
- 使用更高饱和的大面积彩色块。视觉冲击更强，但对终端长时间阅读不友好，也更容易受终端主题差异影响。

### 4. 设计语言采用“训练工作台”而不是工程仪表盘
这套终端 UI 不是通用开发者控制台，而是面向刷题训练的工作台。文案语气保持冷静专业，层级采用中等面板感，状态表达以文字标签为主并辅以少量符号，如焦点指示符、告警提示符和轻量分隔点。这样既能快速扫读，也不会让界面过于嘈杂或像运维面板。

备选方案：
- 工程控制台风格。信息密度高，但容易掩盖“训练上下文”和复盘导向。
- 极简文本风格。更轻，但不足以支撑多工作台、多模式下的稳定层级。

### 5. 帮助与导航约定做成共享行为，而不是各页独立拼接 footer
各页面保留现有核心键位，但统一支持 `j/k` 上下移动和 `?` 帮助切换。默认 footer 只展示核心动作；开启帮助后，页面主体或摘要区展示完整键位与图例。这样既不牺牲常态简洁度，也不要求用户记忆隐藏操作。

备选方案：
- 只在 footer 静态追加所有键位。实现最简单，但信息过密，且页面一多时可读性下降。
- 为每页单独设计帮助逻辑。灵活但重复，后期难以保持一致。

### 6. 统一“上下文 + 主体 + 详情/摘要 + 操作”的阅读结构
selector、sync 详情、browse、stats/review 都采用接近的阅读顺序。根视图优先左侧列表、右侧详情；selector 和 sync 详情用顶部问题/文件上下文，主体左侧表格、右侧摘要；stats 用顶部总览、主体统计或建议、底部操作区。该结构与现有 domain 输出相容，不需要引入新的状态模型。

备选方案：
- 每个页面保留独立布局，仅共享颜色。实现风险较低，但无法真正解决“信息组织不符合阅读习惯”的目标。

### 7. 工作流快捷键采用统一层级语义
`Enter` 统一表示“进入下一层”或“确认当前焦点动作”；`Esc` 统一表示“返回上一层”，只有在根视图才退出当前工作台；`q` 统一表示“直接退出当前工作台”；`Tab` 用于同层模式或页签切换；`b` 仅保留为返回兼容别名，不再作为主提示键位。`sync` 预览页允许安全快操，但复杂的 submission 绑定仍通过详情页完成。

备选方案：
- 保留当前各页面各自定义的返回和模式键。兼容成本低，但用户心智会继续碎片化。
- 让预览页承担完整快操，包括直接选择 submission。交互效率更高，但会把复杂选择压进列表页，削弱可读性与防错性。

## Risks / Trade-offs

- [Risk] 页面拆分后共享 helper 设计不当，可能让不同页面被过度抽象绑定。
  → Mitigation: `common.rs` 只承载明显复用的主题/布局 helper，不抽象页面特有状态机。

- [Risk] 新增 `j/k` 与 `?` 后，键位处理分支变多，容易遗漏已有行为。
  → Mitigation: 把导航和帮助切换提炼为纯函数，并补充键位兼容测试。

- [Risk] 更明显的颜色和高亮在不同终端主题下表现不完全一致。
  → Mitigation: 采用保守的前景/背景语义色和粗体组合，不依赖极端对比或 24-bit 颜色。

- [Risk] 统一 `Esc` 返回后，会与当前 `sync` 中“Esc 等于 skip”的旧语义冲突。
  → Mitigation: 把 `skip` 改为显式动作键，并在帮助与测试中固定“Esc 返回、q 退出、s skip”的新工作流。

- [Risk] 兼容层与新模块并存期间，接口转发可能形成重复入口。
  → Mitigation: 让 `src/tui.rs` 只做薄转发，不保留任何独立逻辑，并在模块注释中标明兼容用途。

- [Risk] 过早引入 onboarding 会让主工作流和教程工作流同时演化，维护成本上升。
  → Mitigation: 本次只把帮助面板和空状态做好，不创建单独 onboarding 能力；待主工作流稳定后再单独设计。
