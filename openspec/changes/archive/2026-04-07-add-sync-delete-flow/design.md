## Context

当前 `sync` 只会收集“有哪些变更文件”，不会区分新增、修改还是删除。后续主流程对每个路径统一执行“提题号 -> 拉题目元数据 -> 拉提交记录 -> 打开提交记录选择界面 -> 生成 commit”的链路，因此删除题目文件也会被误送进 submission/chore/skip 语义里。

这与产品语义不一致。删除题目文件代表维护本地工作区中的文件，不代表删除做题记录，也不天然对应某条在线提交记录。本次设计需要把“删除”从“做题记录绑定”中拆出来，同时保留题目上下文，让 commit 历史仍然可读。

## Goals / Non-Goals

**Goals:**
- 让 `sync` 能识别删除文件，并为删除文件走独立处理分支
- 保持删除操作的语义为“维护文件”，而不是“做题记录选择”
- 为删除文件提供显式确认或跳过的专用交互
- 保留题号和题名等题目上下文，用于生成可读的删除类 commit

**Non-Goals:**
- 不改变普通新增/修改文件的 submission/chore/skip 交互
- 不调整洛谷 API 客户端、缓存 TTL 或元数据解析格式
- 不在本次设计中引入批量删除确认或自动提交删除

## Decisions

### 1. 将变更收集结果从“路径列表”提升为“路径 + 变更类型”

`sync` 主流程要在读取上下文前决定是走“做题记录绑定”还是“删除维护”，因此 `collect_changed_problem_files` 不能再只返回 `Vec<String>`。它需要返回一个显式的变更描述结构，至少区分：

- `Deleted`
- `Active`（覆盖当前的新增/修改/重命名后仍存在的文件）

这样 `cli` 可以在进入 API 和 TUI 前完成分支，避免把删除文件继续送进提交记录查询。

备选方案：
- 继续只返回路径，在 `cli` 中重新检查文件是否存在
  - 否决原因：把 VCS 语义泄漏给调用方，且会把 rename、snapshot 状态等判断分散到主流程里

### 2. 删除文件保留题目元数据，但不查询提交记录

删除文件的 commit 仍需要带题目上下文，因此删除分支继续做题号提取并请求题目元数据；但不会请求 `record/list`，也不会展示 submission 列表。

删除路径的交互模型固定为：
- `confirm delete` -> 创建删除类 commit
- `skip` -> 不生成 commit

这让删除动作保留上下文信息，同时避免把“维护文件”伪装成“绑定线上记录”。

备选方案：
- 删除流程完全不请求元数据，只保留题号和文件名
  - 否决原因：丢失题名后 commit 可读性下降，且当前已有元数据获取路径可复用
- 删除流程继续请求提交记录
  - 否决原因：没有产品价值，只会制造语义混淆和额外网络依赖

### 3. 为删除确认引入独立选择结果和专用 TUI

现有 `SyncSelection` 只表达 `Submission`、`Chore`、`Skip`。删除动作不应复用 `Chore`，否则主流程和 commit message 生成仍会把删除视为本地修改兜底。

因此需要：
- 为模型新增删除确认结果，例如 `Delete`
- 为 TUI 新增专用删除确认入口，Header 继续展示题目信息，主体展示“检测到文件删除”的说明，Footer 只暴露确认删除与跳过操作
- `build_commit_message` 为删除结果生成独立 message 模板，语义固定为“移除题目文件”

备选方案：
- 在现有 submission TUI 中塞入删除分支
  - 否决原因：界面骨架和操作提示会持续混杂“提交记录选择”和“删除确认”两种不同语义

## Risks / Trade-offs

- [风险] `jj diff` 的删除判断若处理不完整，可能误把某些路径继续送入普通分支
  → Mitigation：在变更收集层统一判定删除类型，并用测试覆盖删除与非删除两类输出

- [风险] 删除路径继续拉元数据时，网络失败会影响删除确认体验
  → Mitigation：沿用现有元数据获取策略；若元数据缺失，仍允许基于题号进入删除确认和生成 commit

- [风险] 新增一个专用 TUI 会让键位逻辑多一套分支
  → Mitigation：将删除确认界面限制为最小交互面，只支持确认和跳过，避免复用复杂表格逻辑

## Migration Plan

该变更只影响本地 CLI 的交互和 commit 语义，不涉及数据迁移。发布说明需要明确：从该版本开始，删除题目文件不会进入提交记录选择，而会进入单独的删除确认流程，并生成维护类删除 commit。

## Open Questions

无。本次变更已固定为“删除是文件维护动作；保留题目上下文；专用确认界面；不查询提交记录”。
