## Context

当前 `aclog` 的核心流程集中在 `sync`：它通过 `jj` 识别工作区变更、拉取洛谷题目与提交记录、让用户做显式选择，并为每个文件生成 commit。项目中已经有稳定的 CLI 分发层、`jj` 读取能力、以及基于 `ratatui` 的阻塞式终端界面，但没有用于浏览历史训练结果的独立入口。

这次改动跨越 CLI、版本历史读取、统计聚合和 TUI 渲染多个模块，适合先把实现边界固定下来。项目约束也很明确：不要改动现有 `sync` 行为，不重新抓远端全量做题数据，不重新定义洛谷状态或难度硬编码。

## Goals / Non-Goals

**Goals:**
- 提供独立的 `aclog stats` 命令，让用户在任意时刻查看当前工作区的本地做题统计。
- 基于本地 `jj` 历史中的 `solve(...)` commit 生成统计结果，并在需要算法标签分类时复用或刷新 `/_lfe/tags` 缓存。
- 同时输出“唯一题目口径”和“全部 `solve` 记录口径”，避免同题多次提交导致语义混淆。
- 展示按算法标签聚合的分布，帮助用户判断训练主题覆盖情况。
- 复用现有 TUI 运行框架，提供单屏只读概览和明确空状态。

**Non-Goals:**
- 不实现远端洛谷账号的全量做题统计。
- 不修改 `sync` 的提交流程、交互顺序或现有 commit message 格式。
- 不在首版提供时间趋势图、题目明细浏览、筛选或导出功能。

## Decisions

### 1. 统计数据来源于本地 `jj` 历史中的 `solve(...)` commit

`sync` 已经把已绑定的做题信息写入 commit message，这些数据天然和当前工作区绑定，是统计主体来源。相比重新请求 `/record/list` 或维护新的本地索引，这种方式更稳、实现面更小。算法标签分类则额外依赖 `/_lfe/tags` 提供的标签类型字典，但不改变“做题记录来自本地历史”的主数据源。

备选方案：
- 直接抓取远端洛谷全量记录：信息更全，但会引入新的远端抓取、分页、缓存和口径问题。
- 在 `sync` 时另写统计缓存：读取更快，但会引入额外状态同步和迁移问题。

### 2. 以 commit message 作为统计解析的规范输入

统计逻辑只识别项目当前生成的 `solve(...)` commit message 结构，解析 `Verdict`、`Difficulty`、`Tags`、`Submission-Time` 和 `File` 等字段。遇到缺失字段时回退为 `-`，遇到非 `solve(...)` 或不匹配格式的提交则忽略。

这样可以避免直接依赖 `jj` commit description 之外的内部结构，也不需要新增历史迁移逻辑。旧的手工提交或外部导入提交保持“尽力识别，识别不了就跳过”的策略。

### 3. `/_lfe/tags` 缓存需要保留标签类型，并在统计阶段过滤为算法标签

`/_lfe/tags` 的真实返回为每个标签提供了 `type` 和 `parent`。实现上应把标签缓存从 `id -> name` 升级为 `id -> { name, type, parent }`，并将其持久化到 `.aclog/luogu-tags.toml`。`stats` 聚合时仅保留 `type == 2` 的标签参与统计；`ProblemMetadata.tags` 和 `solve(...)` commit 中的 `Tags:` 字段继续保留原始标签集合，避免丢失来源、时间等原始上下文。

当本地标签缓存不存在或过期时，`stats` 允许读取配置并请求 `/_lfe/tags` 刷新该字典；这属于标签分类辅助数据的同步，而不是远端做题记录同步。

备选方案：
- 在元数据阶段直接过滤：会让 commit message 丢掉原始标签，不符合保真要求。
- 根据名称或父节点猜测标签类别：不稳定，且接口已经给出明确 `type` 字段，没有必要猜测。

### 4. 聚合层同时维护两套口径

统计层会先收集全部 `solve` 记录，再按 `problem_id` 聚合出“唯一题目视图”。其中：
- `solve` 记录数：统计全部 `solve(...)` commit，用于反映训练活动量。
- 唯一题目数：按题号分组后取最新一条记录，作为该题当前状态。
- 难度分布和标签分布：基于唯一题目视图统计，避免同一道题因重复提交重复放大题目属性。

“最新”以历史遍历时的提交时间顺序为准；若无法从 commit 里获得可比较时间，则使用遍历顺序的最后一条作为最新值。这样能把“重做同一题”的场景稳定折叠到唯一题目视图中。

备选方案：
- 只做去重统计：会丢失训练活动量。
- 完全不去重：会让“做过多少题”失真。

### 5. 统计页复用现有 `tui::run_in_terminal()`，首版采用单屏概览

当前 `tui.rs` 已经有可靠的 alternate screen 和 raw mode 包装，统计页可直接复用这层终端管理，只新增只读渲染函数。首版采用单屏布局，显示：
- 标题与样本说明
- 核心计数卡片
- verdict 分布
- difficulty 分布
- tag 分布
- 退出提示

不引入分页、滚动或复杂状态机，避免把一个纯浏览界面做成新的交互系统。

## Risks / Trade-offs

- [历史提交格式不一致] -> 只承诺识别项目当前生成的 `solve(...)` message，无法解析的历史提交直接忽略，并在实现中为解析函数补测试。
- [`stats` 首次运行依赖标签字典] -> 优先使用 `.aclog/luogu-tags.toml`，缺失时再请求 `/_lfe/tags` 并回写缓存，避免每次统计都联网。
- [`jj` 历史读取复杂度增加] -> 把“读取提交描述”和“解析/聚合统计”分层，先在 `vcs` 提供只读历史接口，再在独立模型层做纯数据计算，降低耦合。
- [唯一题目“最新记录”判定依赖历史顺序] -> 优先使用可提取的提交时间；若缺失则在设计中明确回退到遍历顺序，保持行为可预测。
- [单屏界面信息密度有限] -> 首版只展示高价值摘要，把趋势和明细留到后续变更，避免 TUI 复杂度过早膨胀。
