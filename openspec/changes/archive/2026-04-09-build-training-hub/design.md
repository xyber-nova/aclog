## Context

当前 `aclog` 已经有一条可用的记录链路：`sync` 从工作区变更中识别题目文件，结合题目元数据和 submission 生成 `solve(...)` / `chore(...)` / `remove(...)` commit；`record` 负责补录、重绑和列表查看；`stats` 基于本地 `solve(...)` 历史生成首版聚合统计。现有实现的短板也很集中：

- `sync` 是逐文件串行决策，缺少批量预览、dry-run、恢复和防错提示。
- `solve(...)` 记录目前只保留 OJ 元数据，缺少用户自己的训练上下文。
- `record list` 只能输出最新状态，无法支撑复盘、筛选和钻取。
- `stats` 偏静态汇总，不能直接回答“接下来该复习什么”。
- 终端界面目前是若干独立选择器，还不是浏览与复盘工作台。

这次改动会同时触及 CLI、应用层 workflow、commit message 解析、TUI、统计聚合和工作区缓存，是明显的跨模块演进。设计重点是：在不破坏现有 commit 历史兼容性的前提下，把单次记录提升为可浏览、可编辑、可决策的训练资产。

## Goals / Non-Goals

**Goals:**
- 扩展 `solve(...)` 记录格式，增加可选训练字段，并保持旧历史可读。
- 为 `sync` 增加批量预览、dry-run、恢复和一致性提示，同时继续保持显式决策边界。
- 把历史读取抽成统一的记录索引层，供 `record list`、浏览工作台、`stats` 和建议逻辑共享。
- 提供可复用的记录浏览 / 复盘 TUI，并保留等价 CLI 入口。
- 在 `stats` 之上构建 deterministic 的复习候选与训练建议，而不依赖远端服务或黑盒打分。

**Non-Goals:**
- 不引入数据库或独立持久化服务；训练记录仍以 `jj` 历史为事实源。
- 不改变 `chore(...)` / `remove(...)` 的基本语义，也不取消 `sync` 中的显式用户决策。
- 不把 TUI 做成唯一入口；关键能力必须有非交互 CLI 表达。
- 不处理跨题重命名迁移、跨 OJ 聚合或云同步。

## Decisions

### 1. `solve(...)` 记录继续使用可追加的键值正文格式，并将训练字段建模为可选字段

现有 `solve(...)` commit message 已经是稳定的“首行 + Key: Value 正文”结构，解析器也按字段名宽松提取。新的训练信息将继续追加在正文中，采用固定字段名：

- `Note`
- `Mistakes`
- `Insight`
- `Confidence`
- `Source-Kind`
- `Time-Spent`

对应规则：

- 所有新字段都是可选字段，缺失时解析为空或 `None`。
- 旧记录不需要迁移；解析时按“缺失即空”处理。
- `rebind` 默认保留既有训练字段，只更新 submission 相关字段和基于 metadata 重建的题目信息。
- `record edit` 只重写训练字段，不修改题目文件内容。

这样可以最大化复用当前 `commit_format` 和历史遍历逻辑，同时避免引入第二份侧边缓存作为事实源。

备选方案：
- 训练字段写入 `.aclog/*.toml`：编辑方便，但会制造与 `jj` 历史脱节的第二事实源。
- 新增专用 `annotate(...)` commit：实现简单，但会把训练备注和做题记录拆成两套历史，增加统计与浏览复杂度。

### 2. 引入统一的“记录索引层”，把历史解析结果提升为共享读模型

当前 `record list` 和 `stats` 分别从 `jj` 历史拉 message 再各自聚合。后续浏览、过滤、时间线、建议和一致性检查都需要更完整的读模型，因此需要一个统一的记录索引层，负责：

- 读取 `solve(...)` 历史和 revision
- 解析标准字段与新增训练字段
- 生成三类视图：
  - 按文件的当前状态
  - 按题目的当前状态
  - 按文件或题目的完整时间线

该层仍然建立在 `RepoGateway` + `commit_format` 之上，不引入新的存储，只把现有读取能力组合成稳定的领域视图。`record list`、浏览 TUI、`stats` 和建议逻辑都从这个共享视图取数，避免多处重复定义“最新记录”“首次 AC”“重复练习”等口径。

备选方案：
- 每个命令自己读取并聚合：短期改动小，但后续口径必然漂移。
- 额外预构建索引缓存文件：性能更好，但会引入缓存一致性问题；当前数据规模下没有必要。

### 3. `sync` 增加显式的批次会话文件，用工作区内状态支持预览与恢复

`sync` 的“恢复”不能依赖内存，因此会在 `.aclog` 下新增会话文件，例如 `.aclog/sync-session.toml`，记录当前批次：

- 批次创建时间
- 待处理文件清单及其变更类型
- 每项的当前状态：pending / planned / skipped / committed
- 已选择的 action 摘要（submission id、chore、delete、skip）

流程约束：

- `sync --dry-run` 只构建批次和渲染预览，不写 commit，也不保留恢复会话。
- 正常 `sync` 启动时先收集变更并创建或更新批次会话，然后进入预览页。
- 每完成一项显式选择，就刷新会话文件。
- 全部提交成功后删除会话文件。
- 如果发现未完成会话，默认询问继续恢复还是重建批次；CLI 需要提供显式参数跳过恢复。

这样可以把“恢复”做成可验证、可离线测试的工作区行为，而不是临时交互状态。

备选方案：
- 仅保留内存状态：无法跨中断恢复。
- 直接在 commit 创建后推断剩余项：对 skip / chore / delete 的已决状态不稳定，也难以做 dry-run 预览。

### 4. 浏览工作台和统计工作台复用同一套查询结果，TUI 只负责视图切换和交互

当前 `UserInterface` 只暴露选择器与 `show_stats`。为了支持浏览、钻取、编辑和建议展示，需要把 UI 抽象扩展成几类工作台入口：

- sync 批量预览与冲突提示
- record 浏览器（文件视角 / 题目视角 / 时间线视角）
- stats / review 仪表盘
- 可选的训练字段编辑表单

命令层仍负责准备候选集合、过滤结果和默认选中项；TUI 只负责渲染和返回用户动作。等价的 CLI 模式使用相同的查询层和渲染前 DTO，避免 TUI 拥有独占业务逻辑。

备选方案：
- 继续把每个界面写成独立 workflow：会导致数据准备和筛选逻辑散落在 TUI 层。
- 让 TUI 直接查询 API 或 `jj`：违反当前项目已经明确的交互边界。

### 5. 训练建议保持 deterministic，本地可解释，不引入黑盒推荐

复习建议会完全基于本地记录索引和统计口径生成，先实现规则驱动排序，而不是引入复杂模型。首版建议分三类：

- `stale`: 做过但长时间未复习的题
- `weakness`: 最近窗口内高频出现的非 AC / 低熟练度标签
- `retry`: 有明确错误记录、曾多次非 AC 或当前状态非 AC 的题

规则使用可解释信号：

- 最近提交时间
- 首次 AC 与当前状态
- 历史尝试次数
- `Confidence`
- 是否有 `Mistakes`
- 算法标签与时间窗口分布

建议输出必须能解释“为什么推荐这题 / 这个标签”，并且 CLI 与 TUI 看到同一结果。

备选方案：
- 只显示静态统计，不生成候选：实现简单，但无法形成训练决策闭环。
- 引入复杂权重配置：可调性高，但首版会把产品问题转成参数问题，收益不高。

### 6. CLI 命令面扩展以“补全现有对象模型”为准，不新造第二套对象语义

为了覆盖全部能力，命令面按现有对象模型扩展：

- `sync`
  - 增加 `--dry-run`
  - 增加恢复控制参数
- `record list`
  - 增加过滤参数和结构化输出
- `record show <file>`
  - 展示指定文件的当前记录或指定 revision 的完整记录详情
- `record edit <file>`
  - 通过 `--record-rev` 选定目标记录，更新训练字段
- `stats`
  - 增加时间窗口参数
  - 增加建议 / 复习视图入口

这样可以把“记录维护”和“训练浏览”都继续建立在“文件级记录”这个稳定主对象上，而不是引入抽象的、与现有历史结构脱节的新实体。

## Risks / Trade-offs

- [扩展 commit message 字段会增加解析复杂度] → 统一在 `commit_format` 中维护字段常量和解析规则，并补充新旧格式兼容测试。
- [浏览、统计、建议共用索引后会放大口径错误的影响] → 先集中抽象共享读模型，再以单元测试固定“最新记录”“首次 AC”“重复练习”等核心判定。
- [sync 会话文件可能与真实工作区变更漂移] → 恢复前重新校验文件是否仍存在、题号是否仍可解析、待处理项是否仍在变更集中；失效项显式标记而不是静默继续。
- [更强 TUI 会让界面代码快速膨胀] → 先抽公共的渲染前 view model 和页面状态，再分 sync / browser / stats 三个页面族实现。
- [一次覆盖多条能力线，任务量大] → 在 `tasks.md` 中按记录模型、sync、record/browser、stats/review、测试与迁移分组拆解，保证可分阶段实施。

## Migration Plan

1. 先扩展记录模型与解析器，确保旧历史继续可读。
2. 再引入共享记录索引层，并让 `record list` / `stats` 迁移到新读模型。
3. 在此基础上扩展 `sync` 批次预览、恢复和校验。
4. 增加 `record show` / `record edit` 以及浏览工作台。
5. 最后补训练建议与 review 仪表盘。

回滚策略：

- 如果新字段或建议逻辑存在问题，回滚实现即可；旧 `solve(...)` 历史不受影响。
- 会话文件只是工作区缓存，删除 `.aclog/sync-session.toml` 即可回退到无恢复状态。
