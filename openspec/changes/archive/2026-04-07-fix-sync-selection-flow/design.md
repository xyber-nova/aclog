## Context

当前 `sync` 流程会为每个变更文件拉取题目元数据和洛谷提交记录，再调用 TUI 让用户选择对应 submission。问题出在空记录分支：选择器在提交列表为空时直接返回 `Chore`，导致主流程无法区分“用户明确选择 chore”和“系统因为没有记录而自动降级为 chore”。

这个问题不是单纯的文案偏差，而是交互边界定义错误。`sync` 的主语应当是“关联线上提交”，`chore` 只是用户在该语义下的明确兜底，而不是无记录时的默认落点。

## Goals / Non-Goals

**Goals:**
- 统一 `sync` 的选择入口，让所有变更文件都经过同一种交互决策流程
- 保证 `chore` 只来自显式用户操作
- 为无提交记录场景提供明确、可理解的空状态界面
- 保持现有单文件单 commit 的处理模型不变

**Non-Goals:**
- 不改变洛谷 API 获取策略或提交记录解析逻辑
- 不新增“多条 submission 绑定一个文件”或“一个 submission 绑定多个文件”的模型
- 不在本次变更中扩展网络失败、限流、认证失效等异常态交互

## Decisions

### 1. 选择器负责返回用户决策，不负责替用户做默认决策

将 TUI 选择器定义为“用户决策收集器”，而不是“根据数据情况给出默认选择”的组件。

- 选择器有记录时：展示记录列表，支持 `Enter = 选中 submission`、`c = chore`、`Esc = skip`
- 选择器无记录时：展示空状态说明，不预选 submission，但仍支持 `c = chore`、`Esc = skip`

这样可以把 `Submission`、`Chore`、`Skip` 三种结果都收敛为显式决策。

备选方案：
- 保持空记录时直接返回 `Chore`
  - 否决原因：继续混淆“没有记录”和“用户决定记为本地修改”
- 空记录时直接 `Skip`
  - 否决原因：把本应由用户决定的 chore 路径直接抹掉

### 2. `sync` 主流程只消费选择结果，不再推断 chore

`run_sync` 继续负责收集上下文、调用选择器、生成 commit plan，但不再从“空列表”或“空 selection”推导出 `chore`。提交消息构造的语义也同步收紧：

- `Submission(record)` -> `solve(...)`
- `Chore` -> `chore(...)`
- `Skip` -> 不进入 commit plan

备选方案：
- 在 `build_commit_message()` 中保留“非 submission 即 chore”的兜底
  - 否决原因：会重新引入隐式降级路径，掩盖上游流程错误

### 3. 空记录使用同一界面骨架，而不是单独的非交互分支

无提交记录场景仍使用现有选择器入口，减少 `cli` 和 `tui` 之间的分叉逻辑。界面主体显示空状态提示，例如“未找到该题的提交记录”，底部继续展示可用按键。

这样做的好处是：
- 用户始终理解为“`sync` 需要我做一次选择”
- 代码层面无需在 `cli` 里新增第二套分支 UI
- 后续如果要为错误态加 `retry` / `abort`，也更容易沿用同一框架

## Risks / Trade-offs

- [风险] 空状态下没有默认高亮项，键盘交互需要额外处理
  → 规避：将空状态视为“无列表可选”的界面，只响应 `c` 和 `Esc`

- [风险] 现有调用方可能依赖“空记录自动 chore”的旧行为
  → 规避：在 spec 和任务中明确这是一次有意的行为变更，并添加回归测试覆盖

- [风险] 如果未来要支持网络失败交互，当前设计仍需再扩展
  → 规避：本次先统一“空记录”语义，保留选择器作为未来异常态承载点

## Migration Plan

该变更只影响本地 CLI 交互行为，不涉及数据迁移或外部兼容层。发布时需要在变更说明中明确：从该版本开始，无提交记录不会自动创建 `chore` commit，必须由用户显式选择。

## Open Questions

无。本次设计已固定为“发现变更即进入 TUI，用户选择 submission、chore 或 skip”。
