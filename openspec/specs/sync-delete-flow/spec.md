## Purpose

为 `sync` 中的删除类工作区变更定义专门的识别、确认和 commit 语义，避免把文件删除误当成提交记录绑定流程处理。

## Requirements

### Requirement: sync 必须将已删除题目文件与仍存在的变更文件分开分类
当 `aclog sync` 检查工作区变更时，系统 MUST 区分“已删除的题目文件”和“仍存在的变更题目文件”，以便主流程在加载远端记录前选择正确的交互路径。

#### Scenario: 已删除题目文件被识别为删除
- **WHEN** `sync` 检测到某个题目文件路径存在于父树中，但在当前工作副本快照中已不存在
- **THEN** 系统必须将该路径分类为已删除题目文件

#### Scenario: 已修改题目文件不会被识别为删除
- **WHEN** `sync` 检测到某个题目文件发生变更，且该文件在当前工作副本快照中仍然存在
- **THEN** 系统必须将该路径分类为非删除题目文件

### Requirement: sync 必须为已删除题目文件使用专用删除确认流程
当 `sync` 处理已删除题目文件时，系统 MUST 将该动作视为本地文件维护行为，并且不得把该文件送入提交记录选择流程。

#### Scenario: 删除文件跳过提交记录查询
- **WHEN** `sync` 开始处理一个能够提取出题号的已删除题目文件
- **THEN** 系统必须继续加载该题的题目元数据上下文
- **AND** 系统不得为该文件查询用户提交记录列表
- **AND** 系统必须打开专用的删除确认交互，而不是提交记录选择界面

#### Scenario: 删除确认允许显式跳过
- **WHEN** 某个已删除题目文件显示在删除确认交互中，且用户选择跳过该文件
- **THEN** 系统不得为该文件生成 commit

### Requirement: sync 必须为确认删除的文件创建独立的维护类 commit
当用户确认删除某个题目文件时，`sync` MUST 生成删除专用的 commit message，在保留题目上下文的同时明确表达这是一次文件维护动作。

#### Scenario: 确认删除后生成维护类 commit
- **WHEN** 用户为某个已删除题目文件确认删除操作
- **THEN** 系统必须为该文件生成 commit
- **AND** 该 commit message 必须将动作表述为移除题目文件，而不是解题记录或 `chore`

#### Scenario: 删除类 commit 保留题目上下文
- **WHEN** 系统为一个成功识别题号的已删除文件生成删除类 commit
- **THEN** 该 commit message 必须包含题号
- **AND** 该 commit message 必须包含被删除的文件路径
- **AND** 如果题目元数据可用，commit message 必须包含题目标题

### Requirement: 删除确认界面必须只暴露与删除相关的操作
删除确认界面 MUST 将删除动作作为独立交互呈现，其提示文案和按键操作必须符合“维护文件删除”的语义。

#### Scenario: 删除确认界面展示删除专用操作
- **WHEN** 删除确认界面被展示
- **THEN** 界面必须明确提示检测到了文件删除
- **AND** 界面必须提供显式的确认删除操作
- **AND** 界面必须提供跳过操作
- **AND** 界面不得展示提交记录选择说明或 `chore` 操作
